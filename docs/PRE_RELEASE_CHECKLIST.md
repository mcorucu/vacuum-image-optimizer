# Pre-Release Checklist — Vacuum Image Optimizer 0.9.0-rc.1

Run this checklist against a clean WordPress 6.2+ / PHP 8.1+ site before
promoting the Release Candidate. Check each box only after observing the
expected result.

---

## 1. Clean Install
- [ ] Fresh WordPress install with no prior Vacuum data (no `vio_*` options or tables).
- [ ] Upload `vacuum-image-optimizer-0.9.0-rc.1.zip` via **Plugins → Add New → Upload**.
- [ ] Plugin appears in the Plugins list with correct name, version (0.9.0), and description.
- [ ] No PHP notices/warnings on install (with `WP_DEBUG` on).

## 2. Activation
- [ ] Activation succeeds with no fatal error.
- [ ] Custom tables created: `wp_vio_queue`, `wp_vio_stats`, `wp_vio_backups`.
- [ ] Default options created (`vio_settings`, `vio_queue_state`, `vio_db_version`).
- [ ] Daily `vio_cleanup_backups` cron event is scheduled.
- [ ] Activating on PHP < 8.1 is blocked with a clear message (if testable).

## 3. Dashboard
- [ ] **Media → Vacuum Image Optimizer** opens the dashboard without errors.
- [ ] Hero logo and KPI cards render; counts read 0 on a fresh site.
- [ ] All tabs load: Dashboard, Bulk Optimize, WebP & AVIF, Compression, Lazy Load, Backup & Restore, Exclusions, Reports, System Status.
- [ ] System Status shows correct WebP/AVIF/engine availability and writable paths.

## 4. Media Library
- [ ] Upload a JPEG and a PNG.
- [ ] The **Vacuum** column appears and shows "Not optimized".
- [ ] Per-image row actions appear: Optimize, WebP, AVIF, Regenerate, Restore (translatable labels).
- [ ] Clicking **WebP** generates a `.webp` next to the original; column updates to "Optimized" with sizes/savings.
- [ ] A generated WebP (and AVIF, if enabled) appears as a Media Library item with a "Generated by Vacuum Image Optimizer" indicator.
- [ ] Original file is unchanged on disk.

## 5. Queue
- [ ] **Bulk Optimize → Scan Library** queues eligible JPEG/PNG images.
- [ ] **Start Queue** processes in batches; progress updates.
- [ ] **Pause** and **Resume** behave correctly.
- [ ] A forced failure surfaces in **Failed Jobs** with an error message and attempt count.
- [ ] **Retry** re-queues a failed job; after the max (3) attempts it stays failed with the limit reason shown.
- [ ] No image is processed twice when the batch endpoint is triggered concurrently (claim safety).

## 6. Reports
- [ ] **Reports** tab shows totals, storage savings, recent activity, top savings, and format distribution.
- [ ] Figures match the Media Library state.
- [ ] **Export CSV** downloads a file with the documented columns and one row per optimized image.
- [ ] Export completes without memory/timeout errors on a large library (streamed paging).

## 7. Localization
- [ ] **Compression → Interface Language** lists WordPress Default + 10 languages.
- [ ] Selecting a bundled language (e.g. Türkçe, Deutsch) translates the admin UI after save.
- [ ] Tab labels, dashboard cards, and notices are translated.
- [ ] Untranslated strings fall back to English (no missing/blank labels).
- [ ] System Status → Localization shows the text domain loaded and the expected locale.

## 8. Frontend Delivery
- [ ] Enable **Frontend Delivery** with an optimized image and a preferred format.
- [ ] An AVIF-capable browser receives `.avif`; a WebP-only browser receives `.webp`; others get the original.
- [ ] `srcset` candidates are swapped where derivatives exist; no broken image URLs.
- [ ] Stored attachment URLs in the database remain unchanged.
- [ ] Disabling delivery immediately reverts to originals on next load.
- [ ] Native lazy loading adds `loading="lazy"` when enabled.

## 9. Restore
- [ ] With backups enabled, optimize then **Restore** an image.
- [ ] Original is put back; derivative `.webp`/`.avif` files are removed.
- [ ] Generated derivative Media Library items are deleted.
- [ ] Optimization/delivery metadata is cleared and status returns to "Not optimized".
- [ ] Frontend serves the original again (nothing left to deliver).

## 10. Backup Cleanup (retention)
- [ ] With retention = 0, the daily cron runs and deletes nothing.
- [ ] With retention set to N days, expired backups are deleted; originals and active attachments are untouched.
- [ ] `vio_last_backup_cleanup` records examined/deleted/skipped counts.

## 11. Uninstall Test
- [ ] **Deactivate** unschedules the `vio_cleanup_backups` cron.
- [ ] **Delete** runs `uninstall.php`: removes `vio_*` options, custom tables, and `_vio_*` meta.
- [ ] Image files (originals, generated WebP/AVIF, backups) are intentionally left in place.
- [ ] No orphaned cron events or options remain.

## 12. Final Gates
- [ ] `php -l` passes for the main file and all `src/` files.
- [ ] No PHP warnings/notices during a full optimize → report → restore → uninstall cycle with `WP_DEBUG` on.
- [ ] `readme.txt` Stable tag, plugin header Version, and `VIO_VERSION` all read `0.9.0`.
- [ ] Package contains only the files listed in `docs/RC_PACKAGE_STRUCTURE.md`.
